Conception, Choix Technologiques & Paradigmes

La "Golden Stack"

🟢 Backend

NestJS (TypeScript)
Isomorphisme, architecture stricte, écosystème Cloud.

⚛️ Frontend

React (TSX)
Léger, granulaire, Virtual DOM ultra réactif.

🐘 Base de Données

PostgreSQL
Relationnel, intégrité référentielle absolue.

☁️ Stockage

AWS S3
Bande passante infinie, Presigned URLs, auto-purge.

Écosystème & Bibliothèques

🛡️ Zod

Validation isomorphe des données. Un seul schéma partagé (/shared) valide les formulaires React et les requêtes NestJS.

🔷 Prisma

ORM type-safe pour PostgreSQL. Génère automatiquement les types TypeScript depuis le schéma BDD. Migrations versionnées.

📡 Axios

Client HTTP avec intercepteurs. Gestion centralisée du token JWT (injection automatique dans chaque requête via un interceptor).

🐻 Zustand

State management minimaliste (~1 Ko). Remplace Redux sans boilerplate. Idéal pour stocker l'état d'authentification et la liste de fichiers.

🔐 JWT & bcrypt

Authentification stateless (pas de session serveur). Mots de passe hachés avec bcrypt (salt rounds). Token signé côté serveur.

🎨 Tailwind CSS v4

Framework CSS utility-first. Nouveau moteur Oxide ultra-rapide. Zéro fichier de configuration nécessaire en v4.

Justification des Choix (1/2)

  • 🔄 Isomorphisme (NestJS)
    Le choix de NestJS s'impose pour la maintenabilité. Utiliser TypeScript partout permet de partager un dossier /shared. Les schémas de validation Zod sécurisent le Front ET le Back sans aucune duplication de code.
  • ⚛️ Front-end Ciblé (React)
    React a été préféré à Angular car il est beaucoup moins lourd pour une Single Page Application de cette taille. Le Virtual DOM assure des performances optimales pour une interface DataShare qui restera simple. De plus, son vaste écosystème de composants permet un développement ultra-rapide.

Justification des Choix (2/2)

  • ☁️ Délégation S3 (Bande Passante)
    Architecture "Direct-to-S3" via Presigned URLs. Le fichier transite du navigateur directement vers Amazon. Cela protège la carte réseau du serveur (le vrai goulot d'étranglement) et garantit une scalabilité horizontale infinie.
  • 🏗️ Architecture d'Entreprise
    NestJS impose une Inversion de Contrôle et une structure en Services/Controllers, calquées sur Spring Boot, garantissant un code très fortement testable pour valider le document TESTING.md.

Justification des Choix (3/3)

Pourquoi ces bibliothèques et pas d'autres ?

  • 🐻 Zustand plutôt que Redux
    Redux impose un boilerplate considérable (Actions, Reducers, Dispatch). Pour une application de 4 vues avec un seul état global (l'utilisateur connecté + ses fichiers), Zustand offre la même puissance en ~10 lignes de code au lieu de 50.
  • 🔷 Prisma plutôt que TypeORM
    TypeORM souffre de types approximatifs et de bugs non résolus en TypeScript strict. Prisma génère un client 100% type-safe depuis le schéma : toute erreur de requête est détectée à la compilation, pas en production. Les migrations sont versionnées et reproductibles.
  • 🛡️ Zod + Axios : Le contrat blindé
    fetch natif ne fournit aucun typage de réponse. Axios combiné à Zod permet de valider en runtime que la donnée reçue du serveur correspond exactement au schéma attendu. Si l'API change, le front crashe immédiatement en développement, pas silencieusement en production.

Alternatives Écartées (1/2)

  • Java (Spring) / C# (.NET)
    Rois du calcul processeur lourd. Cependant, l'asynchrone est l'ADN par défaut de Node.js, là où il nécessite des modules spécifiques sous Java. Puisque notre architecture S3 délègue l'I/O lourd, l'avantage de l'Isomorphisme (TypeScript) de NestJS a primé.
  • PHP (Laravel)
    Son architecture "Share-Nothing" est très résiliente, mais bloque le thread. L'argument majeur contre PHP ici est la perte de l'Isomorphisme Front/Back (impossibilité absolue de partager le code de validation).

Alternatives Écartées (2/2)

  • Angular
    Surdimensionné. Les Reactive Forms sont excellents, mais le framework pèse trop lourd, pénalisant directement notre budget de performance réseau sur une application de seulement 3 ou 4 vues.
  • Stockage Serveur Local (L'Option B)
    Créerait un backend "Stateful" saturant notre bande passante et empêchant le Load Balancing. Amazon S3 supprime aussi la dette technique liée au développement d'une tâche CRON de purge (US10).

Contrat d'Interface (API REST)

Méthode Route (Endpoint) Auth Payload (Body) Réponse (Success) US
POST /api/auth/register ❌ { email, password } 201 Created US03
POST /api/auth/login ❌ { email, password } { access_token } US04
POST /api/upload/init Optionnel { name, type, size, tags[], pwd?, days? } { fileId, presignedUrl } US01,07-10
PATCH /api/upload/complete/:id Optionnel - 200 OK US01, 07
GET /api/files ✅ JWT - [{ id, name, size, expires_at... }] US05
DELETE /api/files/:id ✅ JWT - 204 No Content US06
POST /api/files/:id/download ❌ { password? } { downloadUrl } (Presigned GET) US02, 09

Isomorphisme : Tous ces payloads (DTOs) seront strictement typés et validés par Zod dans un dossier partagé /shared. React et NestJS utiliseront les mêmes schémas.

Arborescence Backend (NestJS)

backend/
├── prisma/
│   ├── schema.prisma            # Modèles de données (User, File, Tag)
│   └── migrations/
├── src/
│   ├── main.ts                  # Point d'entrée
│   ├── app.module.ts            # Module racine
│   ├── common/                  # 🔧 Guards, Pipes, Decorators, Filters
│   ├── prisma/                  # 🐘 PrismaService (accès BDD injectable)
│   ├── auth/                    # 🔐 module · controller · service · jwt.strategy
│   ├── upload/                  # 🚀 module · controller · service
│   ├── files/                   # 📁 module · controller · service
│   ├── tags/                    # 🏷️ module · controller · service
│   └── s3/                      # ☁️ module · service (AWS SDK)

🧩 1 Module = 1 Domaine
Chaque module encapsule son Controller + Service. Architecture calquée sur Spring Boot.

🔧 common/
Guard (auth JWT), Pipe (validation Zod), Decorator (@CurrentUser), Filter (erreurs).

🐘 Infrastructure injectée
Prisma (BDD) et S3 (cloud) injectés via IoC, jamais accédés directement.

Arborescence Frontend (React)

frontend/src/
├── App.tsx                        # React Router + Layout
├── api/
│   └── client.ts                  # 📡 Instance Axios + Interceptor JWT
├── features/                      # 🧩 Domaines métier
│   ├── auth/                      # 🔐 pages/ (Login, Register) · auth.api · useAuthStore
│   ├── upload/                    # 🚀 pages/ (Upload) · hooks/ (useUpload) · upload.api
│   ├── files/                     # 📁 pages/ (Dashboard, Download) · components/ · useFileStore
│   └── home/                      # 🏠 pages/ (Home)
├── components/                    # 🎨 UI partagés
│   ├── ui/                        # Button, Input, Modal, Badge
│   └── layout/                    # Navbar, Footer, ProtectedRoute
└── utils/                         # 🔧 formatBytes, formatDate

📁 Feature-based
Chaque domaine regroupe pages, composants, store et API. Miroir du backend.

🎨 UI transversaux
Composants réutilisables (Button, Modal, Navbar) partagés entre toutes les features.

📡 Intercepteur centralisé
Un seul client.ts injecte le JWT et gère les 401 (logout auto).
